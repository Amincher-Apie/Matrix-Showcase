## 一、总体架构

刷怪系统由四个层次协作完成：

```
PCG 地图生成 (PcgMapGenerator)
    ↓ PcgMapGenerationResult (含 SpawnPoints)
RunManager 初始化
    ↓
MonsterSpawnManager (服务器权威，Framework/LogicLayer)
    ↓ EnemySpawnService
NetworkObjectPoolManager → 对象池取/还
    ↓
场景中的 EnemyActor 实例 (含 NetworkObject)
    ↓ 死亡事件
EventCenter (UnitDiedEvt) → MonsterRegistry 更新计数
```

### 涉及的文件

| 文件路径 | 职责 |
|---------|------|
| `Framework/LogicLayer/Module/SpawnSystem/MonsterSpawnManager.cs` | **核心：刷怪控制器**，服务器刷怪逻辑主体 |

## 二、生命周期

### 第一阶段：PCG 生成 → 初始化准备

**触发时机**：`RunManager` 收到 `PcgMapGenerator.OnGenerationCompleted` 事件。

**调用链路**：

```
RunManager.OnGenerationCompleted(result, profile)
  └─ monsterSpawnManager.InitializeWithMapResult(result, profile)
```

**`InitializeWithMapResult` 内部流程**：

1. **创建 MonsterRegistry**（单例，全局唯一登记册）
2. **解析可用敌人列表**：
   - 遍历 `profile.AvailableEnemies`（`List<EnemySO>`）
   - 组合为 `"{rank}/{id}"` 格式，如 `"Normal/001"`, `"Elite/002"`
   - 若 profile 为空，回退到 Inspector 上的 `monsterEnemyId`（默认 "001"）
3. **扫描刷怪点**：
   - 遍历 `result.SpawnPoints`
   - 过滤 `Category == NormalEnemy`（排除 BossEnemy / DefenseObjective）
   - 跳过 `PointTransform == null` 的无效点
   - 按 `NodeId`（房间区域）分组存入 `_regionSpawnPoints` 字典
4. **缓存 RoomGraph**：用于后续 BFS 距离计算
5. **计算初始动态上限**：`RefreshDynamicCap()`
6. **启动计时器**：`_isRunning = true`，进入持续刷怪阶段

### 第二阶段：持续刷怪循环

**驱动方式**：`MonsterSpawnManager` 的 `FixedUpdate()`

```
FixedUpdate()
  ├─ PollMissionState()       ← 每帧同步任务/玩家/难度状态
  └─ TickSpawn()              ← 间隔计时，到达后执行 OnSpawnTick()
```

#### 2.1 状态轮询 (`PollMissionState`)

每帧从各单例读取当前状态：

| 数据 | 来源 | 用途 |
|------|------|------|
| `_difficultyTier` | `RunManager.GetDifficultyTier()` | 怪物等级 = 1 + difficultyTier |
| `_currentPlayerCount` | `NetworkManager.ConnectedClientsList.Count` | 动态上限/刷怪速率倍率 |
| `_activeTaskCount` | `MissionManager.RuntimeMissions` 中 `State==Active && Category==Secondary` 的数量 | 任务数倍率 |
| `_occupiedRegions` | 所有 `PlayerActor` 所在房间的 `NodeId` | BFS 起点（兴趣点权重） |

#### 2.2 刷怪 Tick (`OnSpawnTick` 流程)

```
OnSpawnTick():
  1. if (ActiveMonsterCount >= _dynamicCap) → return           ← 已达上限，本次不刷
  2. budget = floor(ComputeCurrentSpawnRate())                 ← 计算单次刷怪预算
  3. if (budget <= 0) → return
  4. regionWeights = ComputeRegionWeights()                    ← 兴趣点算法
  5. allocation = DistributeBudget(budget, regionWeights)      ← 按权重分配预算
  6. foreach (nodeId, count in allocation):
       perPoint = count / points.Count
       remainder = count % points.Count
       foreach (point in points):
         spawn count 个怪物
```

## 三、关键算法

### 3.1 动态上限 (`DynamicCap`)

```
DynamicCap = min(
    baseCapPerRegion × regionCount × playerMult × taskMult,    // 动态计算值
    profile.MonsterSpawn.globalMaxCap,                         // 风格硬上限
    config.globalMaxActiveMonsters                             // 全局硬上限（默认 200）
)
```

- **`baseCapPerRegion`**：来自 `PcgGenerationProfile.MonsterSpawnSettings`，默认 20
- **`playerMult`**：玩家数倍率

| 玩家数 | 1 | 2 | 3 | 4 |
|--------|---|---|---|---|
| Cap 倍率 | 1.0x | 1.5x | 2.0x | 2.5x |
| Rate 倍率 | 1.0x | 1.2x | 1.5x | 2.0x |

- **`taskMult`**：活跃支线任务倍率（线性叠加，有上限）

```
taskMult = min(
    firstTaskCapMultiplier + (taskCount - 1) × additionalTaskCapMultiplier,
    maxTaskCapMultiplier
)
// 默认：1.5 + (n-1) × 0.5, 上限 3.0
```

### 3.2 刷怪速率 (`SpawnRate`)

```
每 Tick 预算 = baseSpawnRatePerTick × playerRateMult × taskRateMult
```

- `baseSpawnRatePerTick`：来自 profile，默认 3
- 默认 `spawnTickInterval`: 2 秒
- 速率任务倍率：`first * 2.0 + (n-1) * 1.0`, 上限 4.0

### 3.3 兴趣点算法 (`ComputeRegionWeights`)

**核心思想**：怪物刷在离玩家适当距离的房间——太近危险，太远没交互。

```
权重表（默认）:
  距离 1（邻接房间）   → 0.5    ← 少量，玩家进门前可能撞上
  距离 2（隔一间）     → 1.0    ← 主要刷怪区
  距离 3（隔两间）     → 0.5    ← 少量，探索方向
  距离 ≥ 4             → 0.0    ← 不刷（太远，玩家不会去）
  距离 0（玩家所在）   → 0.0    ← 不刷（玩家当前房间）
```

**计算步骤**：

1. 从 `_occupiedRegions`（所有玩家所在 NodeId）出发
2. 对每个有刷怪点的区域，用 BFS 求到最近玩家的最短距离
3. 距离 → 权重映射（可配置于 `MonsterSpawnConfig`）
4. 加权随机分配预算（`DistributeBudget` 使用 floor 分配 + 余数投到权重最大区域）

> 算法依赖 `RoomGraph.GetShortestDistance()`（BFS 实现于 `RoomGraphExtensions.cs`）

### 3.4 预算分配 (`DistributeBudget`)

```
1. 按 ratio_i = weight_i / totalWeight 计算每个区域的整数分配（floor）
2. 计算已分配总数与 budget 的差值（remainder）
3. 将 remainder 全部分给权重最高的区域
```

这种分配策略保证预算不会浪费，且自然向高权重区域倾斜。

## 四、刷怪执行细节

### 4.1 单次刷怪 (`SpawnMonsterAt`)

```
SpawnMonsterAt(pointData):
  1. if (ActiveMonsterCount >= _dynamicCap) → return              ← 二次检查
  2. enemyId = random(_availableEnemyIds)                         ← 随机选敌人类型
  3. monsterNetObj = enemySpawnService.SpawnEnemy(enemyId, pos)   ← 从对象池取
  4. monsterLevel = 1 + _difficultyTier                           ← 难度等级 → 怪物等级
  5. serverAttr.SetLevelServerRpc(monsterLevel)                   ← 同步等级
  6. MonsterRegistry.Register(monsterNetObj)                      ← 登记
  7. OnEnemySpawned?.Invoke(...)                                  ← 回调节点
```

### 4.2 EnemySpawnService 封装

`EnemySpawnService.SpawnEnemy()` 内部：
1. 拼接路径 `"Prefab/Enemy/{rank}/{id}"`
2. 调用 `NetworkObjectPoolManager.GetAndSpawn(prefabPath, pos, rot, beforeSpawn)`
3. `beforeSpawn` 回调中：
   - 调用 `EnemyActor.ConfigureForSpawn(enemyId, prefabPath, aiConfigPath)`
   - 调用 `EnemyActor.ActivateAfterSpawn()`
   - 设置 `ServerEnemyAttributeModule.SetPrefabPath(prefabPath)`

### 4.3 怪物等级

- `_difficultyTier` 来自 `RunManager.GetDifficultyTier()`（根据对局已用时间计算）
- 怪物等级 = `1 + _difficultyTier`
- 通过 `ServerRpc` 同步到客户端

## 五、死亡追踪

**不使用** Update 中轮询，而是通过 `EventCenter` 事件驱动：

```
EventCenter.UnitDied (UnitDiedEvt)
  └─ MonsterRegistry.ReportKillByNetworkId(evt.unitId)
       └─ 遍历字典，按 NetworkObjectId 匹配 → 移除 + killed++
       └─ 同时触发 OnEnemyDespawned 回调（供 RunManager 做 TeamWipe 判定）
```

**MonsterRegistry** 维护三个计数：
| 属性 | 含义 |
|------|------|
| `ActiveMonsterCount` | 当前活跃数（字典大小） |
| `TotalSpawnedCount` | 累计生成数（仅增不减） |
| `TotalKilledCount` | 累计击杀数（仅增不减） |

## 六、难度等级系统

### 6.1 难度等级计算

由 `RunManager` 提供：

```
difficultyTier = floor(runElapsedMinutes / intervalMinutes)
默认 interval = 3 分钟，每 3 分钟升一级
```

### 6.2 难度等级的影响

| 影响项 | 作用方式 | 代码位置 |
|--------|---------|---------|
| **怪物等级** | 怪物等级 = 1 + difficultyTier | `SpawnMonsterAt()` L407 |
| **动态上限** | `_difficultyTier` 变更时重算 `RefreshDynamicCap()` | `PollMissionState()` L459 |

> 注意：当前实现中 `_difficultyTier` 被读取但没有直接作用于 cap 计算公式。它在 `PollMissionState` 末尾触发了 `RefreshDynamicCap()`，但如果 cap 公式本身不依赖 `_difficultyTier`，则实际上只会因为 `_currentPlayerCount` 或 `_activeTaskCount` 的变化才触发重算。**需要人工确认**是否应在 cap 公式中加入难度等级倍率。

## 八、与 RunManager 的交互

### 接口维度

| 交互 | 方向 | 方式 |
|------|------|------|
| PCG 结果注入 | RunManager → SpawnManager | `InitializeWithMapResult()` |
| 难度等级查询 | SpawnManager → RunManager | `FindObjectOfType<RunManager>().GetDifficultyTier()` |
| 怪物生成/销毁回调 | SpawnManager → RunManager | `OnEnemySpawned` / `OnEnemyDespawned` 委托 |
| 死亡事件 → TeamWipe 判定 | EventCenter → RunManager | `EventName.UnitDied` 监听 |

### RunManager 状态机中的角色

| RunState | 刷怪行为 |
|----------|---------|
| `RunInit` | 初始化，`_isRunning = false` |
| `RoomEnter` → `Exploring` | **自动运行**，SpawnManager 自主刷怪 |
| `BossFight` | **继续运行**（Boss 战不阻止环境刷怪） |
| `RunVictory / RunDefeat` | `Shutdown()` 停止 |

## 九、与 MissionSystem 的交互

- SpawnManager **不直接**为任务刷怪
- 任务刷怪由 `MissionBase` 及其实现（如 `EliminationMission`）自行调用 `EnemySpawnService`
- SpawnManager 只负责**环境刷怪**（ambient spawn），提供持续的"世界填充"
- 两者共享同一对象池（`NetworkObjectPoolManager`）和死亡事件通道（`EventCenter.UnitDied`）

```
环境刷怪: MonsterSpawnManager (按玩家位置/距离权重)
任务刷怪: MissionBase 实现 (按任务配置，如指定房间/波次)
             ↓                ↓
        EnemySpawnService → NetworkObjectPoolManager
             ↓                ↓
        MonsterRegistry + EventCenter.UnitDied
```

## 十一、数据流图

```
┌─────────────────────────────────────────────────────────┐
│  FixedUpdate (每帧)                                      │
│                                                          │
│  ┌─ PollMissionState() ──────────────────────────────┐   │
│  │  NetworkManager  → _currentPlayerCount             │   │
│  │  MissionManager  → _activeTaskCount                │   │
│  │  PlayerActor 位置 → _occupiedRegions               │   │
│  │  RunManager      → _difficultyTier                 │   │
│  │  └─ RefreshDynamicCap() (状态变更时触发)            │   │
│  └────────────────────────────────────────────────────┘   │
│                                                          │
│  ┌─ TickSpawn() ─────────────────────────────────────┐   │
│  │  定时器累计 fixedDeltaTime                         │   │
│  │  到达间隔 → OnSpawnTick()                          │   │
│  │                                                    │   │
│  │  1. ActiveCount < DynamicCap?                      │   │
│  │     ↓ yes                                          │   │
│  │  2. budget = floor(rate * playerMult * taskMult)   │   │
│  │     ↓ budget > 0?                                  │   │
│  │  3. ComputeRegionWeights():                        │   │
│  │     BFS(玩家位置 → 各区域) → 距离 → 权重           │   │
│  │     ↓ weights > 0?                                 │   │
│  │  4. DistributeBudget(budget, weights)              │   │
│  │     ↓ allocation > 0?                              │   │
│  │  5. SpawnMonsterAt(point) × count                  │   │
│  │     ↓                                              │   │
│  │  6. Registry.Register + OnEnemySpawned             │   │
│  └────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

## 十二、已知限制与待改进

| 问题 | 影响 | 状态 |
|------|------|------|
| `_difficultyTier` 在 cap 公式中无直接作用 | 难度等级仅影响怪物等级，不影响刷怪量 | 待确认是否需要加入公式 |
| `PollMissionState` 每帧 `FindObjectOfType<RunManager>()` | 性能浪费 | 建议缓存引用 |
| `_occupiedRegions` 通过遍历所有 `PlayerActor` 并比对房间包围盒计算 | 大场景可能有性能开销 | 可考虑 PlayerActor 定时广播自己所在 NodeId |
| 权重计算全在刷怪 tick 执行 | 每 2 秒一次 BFS，房间数多时代价较高 | 必要时可缓存 BFS 结果并增量更新 |
| `MonsterRegistry.ReportKillByNetworkId` 使用线性查找 | O(n)，大规模战斗可能有压力 | 建议改为 `Dictionary<ulong, int>` 双索引 |
